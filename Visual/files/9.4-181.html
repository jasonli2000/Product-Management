    34. PATCH APPLICATION HISTORY:   39 SEQ #33
      DATE APPLIED:   2003-10-02 00:00:00
      APPLIED BY:   USER,ONE
      DESCRIPTION:   
      The View Patient Address [EAS VIEW PATIENT ADDRESS] menu option will 
allow a user to display the Patient's address along with Address Change 
Date, Address Change Source, and Address Change Site  (if applicable) 
information.  This option calls the EASAILK1 routine, which does a lookup 
on the patient file, then, calls the standard device API (%ZIS) to open 
the selected output device to output the information to.  This error is 
occurring because immediately after calling the device API (%ZIS), the 
routine invokes the USE command to open the device to begin sending 
output to it.  
 
In the situation where the user queues the output either by entering a 
"Q" at the device prompt, or the selected device has the QUEUEING Field 
(#5.5) set to "FORCED" in the DEVICE File (#3.5), when the USE command is 
invoked, the DEVICE has not yet been opened by the process because it's 
been queued for a later time.  The error results due to the device not 
yet being available.

    40. PATCH APPLICATION HISTORY:   44 SEQ #39
      DATE APPLIED:   2004-03-02 00:00:00
      APPLIED BY:   USER,ONE
      DESCRIPTION:   
      SDC-1003-60902
ATG-0803-32133
 
Under certain conditions, when a user creates a new link a between a new 
web based 1010EZ application in the 1010EZ HOLDING File (#712) and the 
PATIENT File (#2) using the menu protocol Link to Patient File [EAS EZ 
LINK TO FILE 2] of the Electronic 10-10EZ Processing [EAS EZ 1010EZ 
PROCESSING] List Manager application, the incorrect patient is selected 
and displayed.  This is caused by the FileMan DIE call which is used to 
update the LINK TO FILE #2 field (#3.4) in the 1010EZ HOLDING File 
(#712).  Since this field is a pointer to the PATIENT File, when it's 
updated, FileMan calls the patient lookup, DPTLK, to validate the patient 
DFN being passed in.  The patient lookup checks a series of 
cross-references on the patient file, one of which is the "SSN" 
cross-reference.  When the DFN being checked matches the initial portion 
of a SSN uniquely, the IEN of this matching SSN is returned and stored as 
the pointer to the PATIENT File instead of the correct IEN.
